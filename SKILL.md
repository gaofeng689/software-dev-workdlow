# 软件开发工作流 Skill

这是一个完整的软件开发流程指南，遵循"规划 → 执行 → 验证 → 迭代"的开发范式。

## 核心原则

1. **先思后行**：充分理解需求再动手，避免返工
2. **小步快跑**：大任务拆分为小任务，逐步完成
3. **持续验证**：每个节点都要验证，而非最后一次性测试
4. **闭环交付**：确保每个任务真正完成并通过测试

---

## 开发流程

### 第一阶段：头脑风暴

**目标**：充分理解需求，澄清疑问，明确边界

当你收到一个开发需求时，首先进行头脑风暴：

1. **澄清需求**
   - 询问用户想要解决什么问题
   - 确认输入是什么，期望输出是什么
   - 了解使用场景和用户群体
   - 识别潜在约束（如性能要求、兼容性要求、安全要求等）

2. **分析可行性**
   - 技术上是否可行
   - 是否有现成的方案可以参考
   - 需要的依赖和资源

3. **输出**：形成清晰的需求理解文档

**触发示例**：
- "用户要求开发一个用户认证模块"
- "需要实现一个文件上传功能"

---

### 第二阶段：任务定义与拆分

**目标**：将大任务拆分为可管理的小任务

1. **定义任务范围**
   - 明确要做什么（Scope In）
   - 明确不做什么（Scope Out）
   - 设定优先级

2. **拆分任务项**
   - 将大任务拆分为 3-7 个子任务
   - 每个子任务应该可以在较短时间内完成（建议 15-60 分钟）
   - 识别任务之间的依赖关系

3. **定义完成标准（Acceptance Criteria）**
   - 每个子任务必须有明确的完成标准
   - 完成标准要具体、可验证
   - 包括功能性需求和非功能性需求（如性能、可维护性）

**任务拆分示例**：
```
任务：实现用户认证模块
├── 子任务 1：设计用户数据模型 → 完成标准：有用户表结构，包含必要字段
├── 子任务 2：实现注册功能 → 完成标准：能注册用户，密码加密存储，重复注册报错
├── 子任务 3：实现登录功能 → 完成标准：能登录，返回 JWT token
├── 子任务 4：实现登出功能 → 完成标准：token 失效
└── 子任务 5：编写单元测试 → 完成标准：核心功能测试覆盖率达到 80%+
```

---

### 第三阶段：Subagent 决策

**目标**：决定是否启用 Subagent 来并行处理独立任务

1. **分析任务依赖**
   - 哪些任务可以并行执行
   - 哪些任务必须串行执行

2. **Subagent 启用条件**
   - 启用 Subagent 的情况：
     - 多个独立任务可以并行处理
     - 任务涉及不同的技术栈或模块
     - 需要同时进行多个文件的开发
   - 不启用 Subagent 的情况：
     - 任务之间有强依赖关系
     - 需要统一的代码风格或架构决策
     - 调试阶段需要集中处理

3. **任务分配策略**
   - 主 Agent 处理：架构设计、核心逻辑、关键模块集成
   - Subagent 处理：独立的功能模块、辅助工具、测试用例

---

### 第四阶段：开发与迭代

**目标**：按计划执行开发，持续调整优化

1. **编码规范**
   - 遵循项目的代码风格
   - 保持代码简洁、可读
   - 及时添加必要的注释

2. **持续重构**
   - 在开发过程中识别代码坏味道
   - 及时调整代码结构
   - 提取重复代码为公共函数/类

3. **性能优化**
   - 识别性能瓶颈
   - 优化算法复杂度
   - 减少不必要的计算和 I/O

4. **进度跟踪**
   - 每完成一个子任务，记录进度
   - 如遇问题，及时调整计划
   - 保持与用户的沟通

---

### 第五阶段：编译测试

**目标**：验证所有任务是否符合预期

1. **编译检查**
   - 确保代码无语法错误
   - 确保所有依赖正确引入
   - 确保构建成功

2. **单元测试**
   - 运行所有单元测试
   - 确保测试覆盖率符合要求
   - 修复失败的测试用例

3. **集成测试**
   - 验证模块之间的集成正确性
   - 验证端到端功能流程

4. **边界测试**
   - 测试极端情况和边界条件
   - 验证错误处理逻辑

---

### 第六阶段：迭代优化

**目标**：测试失败时持续修改直到通过

1. **问题分析**
   - 明确失败原因
   - 区分：代码错误、测试错误、需求理解错误

2. **修复策略**
   - 代码错误 → 修复代码
   - 测试错误 → 修复测试用例
   - 需求错误 → 回到第一阶段重新澄清

3. **回归测试**
   - 修复后重新运行所有测试
   - 确保修复没有引入新问题

4. **循环迭代**
   - 直到所有测试通过
   - 或与用户确认接受当前状态

---

## 工作流程检查清单

在每个阶段完成后，检查以下问题：

### 头脑风暴阶段
- [ ] 需求是否清晰
- [ ] 是否有歧义需要澄清
- [ ] 技术可行性是否确认

### 任务拆分阶段
- [ ] 任务是否已拆分
- [ ] 每个任务是否有明确的完成标准
- [ ] 任务依赖关系是否明确

### Subagent 决策阶段
- [ ] 是否需要启用 Subagent
- [ ] 任务分配是否合理

### 开发阶段
- [ ] 代码是否符合规范
- [ ] 是否进行了必要的重构
- [ ] 进度是否符合预期

### 测试阶段
- [ ] 编译是否成功
- [ ] 单元测试是否通过
- [ ] 功能是否满足需求

### 迭代阶段
- [ ] 所有测试是否通过
- [ ] 是否有未解决的问题
- [ ] 是否可以交付

## 输出格式

在完成整个工作流后，输出最终报告：

```markdown
# 开发完成报告

## 任务概述
[简要描述完成的任务]

## 完成的任务项
1. [子任务 1] - ✅ 完成
2. [子任务 2] - ✅ 完成
...

## 测试结果
- 单元测试：通过 X 个，失败 0 个
- 集成测试：通过
- 代码覆盖率：XX%

## 遗留问题（如果有）
- [问题 1] - 原因说明
- [问题 2] - 原因说明

## 可选：后续建议
- [建议 1]
- [建议 2]
```

## 注意事项

1. **不要跳步**：严格按照流程执行，不要跳过头脑风暴或任务拆分阶段
2. **及时沟通**：开发过程中有问题及时与用户沟通
3. **保持灵活**：根据实际情况调整流程，不必死板遵循
4. **质量优先**：追求高质量交付，而非快速交付
5. **文档优先**：重要决策和设计应该记录下来

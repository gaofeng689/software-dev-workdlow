# test-assist 功能测试与修复技能

## 核心工作流程

当用户要求测试某个功能时，执行以下流程：

```
功能理解 → 测试计划制定 → 功能验证执行 → 问题识别与修复 → 总结报告
```

---

## 第一步：理解要测试的功能

**必须明确以下信息：**

1. **测试对象**：要测试的具体功能是什么？（如：某个API、某个类、某个页面功能）
2. **测试范围**：功能的哪些方面需要验证？（输入验证、输出正确性、边界条件、错误处理等）
3. **测试环境**：代码位置、依赖环境、运行条件
4. **预期行为**：用户认为这个功能应该怎么工作？

**如果信息不明确，立即向用户确认：**
```
❓ 需要确认：
- 测试的功能具体是什么？（文件路径、类名、方法名）
- 期望的行为是什么？
- 是否有特定的测试场景需要重点验证？
```

---

## 第二步：制定测试计划

基于理解，制定简短的测试计划，包含：

1. **测试场景**：列出将验证的具体场景（如：正常输入、边界值、异常输入）
2. **验证方法**：如何验证每个场景（代码审查、运行测试、执行API调用等）
3. **成功标准**：什么是"测试通过"？

**输出格式：**
```markdown
## 测试计划

### 测试对象
[功能描述和位置]

### 测试场景
1. [场景1描述] - 验证方法：[方法]
2. [场景2描述] - 验证方法：[方法]
...

### 成功标准
- [标准1]
- [标准2]
```

---

## 第三步：执行功能验证

### 3.1 代码审查
- 阅读相关代码，理解实现逻辑
- 识别明显的逻辑错误、边界条件遗漏

### 3.2 动态测试
根据功能类型选择合适的方式：

| 功能类型 | 测试方式 |
|---------|---------|
| API/接口 | 构造请求并调用，观察返回 |
| 类/方法 | 编写临时测试代码调用 |
| 前端功能 | 使用 Playwright 模拟操作 |
| 数据库操作 | 执行SQL验证 |

### 3.3 记录测试结果

```markdown
## 测试执行记录

### 场景1：[描述]
- 测试方法：[使用的验证方法]
- 输入：[测试输入]
- 预期输出：[期望的结果]
- 实际输出：[实际观察到的]
- 结果：✅ 通过 / ❌ 失败
- 问题（如失败）：[发现的问题]
```

---

## 第四步：问题修复

**只有当发现明确问题时才进行修复。**

### 修复原则

1. **最小改动**：只改必要的部分，不做额外重构
2. **不引入新问题**：修复后需重新验证
3. **可逆性**：保留修改前的代码（使用 Edit 工具会自动备份）

### 修复流程

```
问题定位 → 分析原因 → 制定修复方案 → 实施修复 → 回归验证
```

**如果问题无法安全修复：**
```
⚠️ 发现问题但无法安全修复：
- 问题：[描述]
- 原因：[分析]
- 建议：[如何手动修复]
```

---

## 第五步：生成总结报告

测试完成后，生成以下格式的报告：

```markdown
# 功能测试报告

## 测试概要
- 测试对象：[功能名称]
- 测试时间：[时间]
- 测试人员：Claude (test-assist)

## 测试结果
- 总测试场景：N
- 通过：N ✅
- 失败：N ❌

## 发现的问题

### 问题 1：[标题]
- **严重程度**：[高/中/低]
- **位置**：[文件:行号]
- **描述**：[问题描述]
- **修复措施**：[已实施的修复] / [建议修复方式]
- **状态**：[已修复 / 需人工介入]

### 问题 2：...

## 修复验证
[列出所有修复后重新验证的结果]

## 总结
[总体评估和建议]
```

---

## 重要提示

1. **不要臆测**：如果用户描述的功能位置不明确，先确认再测试
2. **最小改动**：修复时只改必要的代码
3. **逐个验证**：每修复一个问题，立即验证该问题是否解决
4. **诚实报告**：如果测试无法完成某项验证，诚实说明原因
5. **安全第一**：危险的修改（如删除文件、修改生产配置）需先确认

# Bug 修复技能

## 核心原则

**不主观臆断，不掩饰困惑，清晰列明取舍。** 修复 bug 需要严谨的排查过程，不能靠猜测。

## 修复流程

### 1. 复现问题（Reproduce）

**首要任务：确认 bug 真实存在并理解其表现。**

- 收集信息：
  - 错误信息/堆栈跟踪（stack trace）
  - 复现步骤（最小化复现用例）
  - 预期行为 vs 实际行为
  - 发生频率（必现/偶发）

- 如果可能，直接运行代码复现问题
- 如果无法复现，询问用户更多上下文

### 2. 定位问题（Locate）

**基于症状推断可能的原因，而非猜测。**

- 分析错误类型：
  - `NullPointerException` → 空指针检查
  - `IndexOutOfBoundsException` → 数组/集合边界
  - `ConcurrentModificationException` → 并发修改问题
  - 业务逻辑错误 → 状态流转/计算逻辑

- 使用工具定位：
  - 添加日志输出关键变量值
  - 使用断点调试
  - 检查最近修改的代码（git blame）

- 缩小范围：
  - 二分查找法：逐步排除可疑代码段
  - 控制变量：确认哪个输入导致问题

### 3. 分析根因（Analyze）

**找到根本原因，而非表象。**

- 问自己：
  - 为什么这里会出错？
  - 这是首次出现还是一直存在？
  - 最近有什么变更导致这个问题？

- 如果是偶发问题：
  - 竞态条件？
  - 缓存问题？
  - 网络超时/重试？

### 4. 修复问题（Fix）

**精准改动，只改必需的部分。**

- 修复策略：
  - 如果原因明确，直接修复
  - 如果原因不明，先添加防御性检查
  - 如果需要重构，先确保有测试覆盖

- 修复时：
  - 只改导致 bug 的代码，不"顺便优化"周边代码
  - 保持代码风格一致
  - 移除修复过程中添加的临时调试代码

- 修复后：
  - 验证修复有效
  - 确认没有引入新问题

### 5. 验证修复（Verify）

**确保修复正确且完整。**

- 用相同步骤再次测试，确认问题不再出现
- 测试边界条件（空值、极端值、并发等）
- 检查相关功能是否受影响
- 如果可能，添加或更新测试用例

## 常见问题处理

### 偶发性 Bug
```
1. 检查日志时序
2. 确认是否与并发/异步有关
3. 检查资源竞争（数据库连接、文件句柄等）
4. 考虑添加详细日志捕获更多信息
```

### 性能相关 Bug
```
1. 使用 profiler 定位热点
2. 检查数据库查询是否需要优化
3. 检查是否有不必要的循环或重复计算
```

### 第三方库 Bug
```
1. 确认问题确实在第三方库而非自己的代码
2. 检查是否有已知的 issue 和 workaround
3. 考虑升级版本或寻找替代方案
```

## 输出格式

修复完成后，提供以下信息：

```
## 问题分析
- 根本原因：[简述根本原因]

## 修复方案
- 修改文件：[列出修改的文件]
- 修改内容：[简述修改]

## 验证结果
- 复现步骤：[用来验证的测试步骤]
- 结果：[是否修复成功]
```

## 注意事项

- **不要猜测**：如果信息不足，明确告诉用户需要什么信息
- **不要过度修复**：只修复报告的问题，不做预防性修改
- **保留证据**：如果添加了调试代码，记录为什么添加，便于后续清理
- **安全第一**：如果 bug 涉及安全漏洞，优先处理安全问题
